Figma Technical Handoff Guide & Best Practices

Moood Studio Agency - Automated Design-to-Code Pipeline

This document outlines the strict technical requirements for Figma files handed off to Moood Studio Agency. Our agency utilizes an advanced AI-powered pipeline (Model Context Protocol) that automatically reads, interprets, and translates your Figma file into code tokens and structure.

To ensure a zero-friction handoff and a pixel-perfect build without manual back-and-forth, your Figma file must adhere to the following best practices.

1. Variables and Modes (The Foundation)

Our system natively understands Figma Variables. Detached hex codes or hardcoded sizes slow down the process and introduce errors.

  • Use Variables for Everything: Colors, Spacing, Border Radii, and Sizing must be defined as variables, not just local styles.
  • Semantic Naming: Name variables by their function, not their appearance.
    • ❌ Bad: Color/Blue or Text/48px
    • ✅ Good: Brand/Primary, Spacing/Section-Gap, Radius/Card
  • Modes for Responsive Design: If design values change between Desktop and Mobile (e.g., paddings or typography scales), map them using Variable Modes (e.g., Mode 1: Desktop, Mode 2: Mobile). Our build agents will automatically read these modes to generate fluid CSS functions (clamp()) and media queries.
  • Modes for Theming: Use Variable Modes for Light and Dark themes instead of duplicating screens.

2. Typography System

We use Fluid Typography. This means text gracefully scales between mobile and desktop sizes mathematically.

  • Strict Text Styles: Every text node must have an assigned Text Style. No detached typography.
  • Clear Hierarchy: Group text styles semantically. (e.g., Heading/Display, Heading/H1, Body/Standard, Body/Small, UI/Label).
  • Mode-Linked Typography: Link numeric variables to your Text Styles. If Heading/H1 is 120px in the "Desktop" mode and 48px in the "Mobile" mode, our system will automatically parse this and generate the perfect CSS clamp() rule.

3. Auto Layout is Mandatory

Our extraction agents read the DOM (Document Object Model) structure directly from your Figma frames. If a layout relies on absolute positioning by dragging and dropping, it cannot be accurately translated to HTML/CSS flexbox or grid.

  • 100% Auto Layout: Every section, container, card, and grouping must use Auto Layout.
  • Resizing Rules: Accurately set elements to Hug contents, Fill container, or Fixed width/height. Our engine translates "Fill container" directly to width: 100%; or flex: 1;.
  • No "Hacks": Avoid using invisible spacer boxes. Use the native Gap, Padding, and Margin properties within Auto Layout.

4. Spacing and Structure

  • Use Spacing Variables: Define a spacing scale (e.g., 4px, 8px, 16px, 24px, 32px, 64px) using number variables. Apply these variables to your Auto Layout gaps and paddings.
  • Logical Grouping: Frame your elements logically. A typical web section should be structured as:

    Section > Container (Max-Width) > Content Layout > Elements

5. Asset Export Configuration

Our agents will automatically download assets directly from the remote Figma file.

  • Mark for Export: Any vector illustration, icon, or photographic asset that needs to exist on the website MUST have an Export setting applied (+ Export in the right panel).
  • Formats:
    • Icons and Logos: Set to SVG.
    • Photos/Complex Graphics: Set to PNG, JPG, or WebP (at @2x minimum).
  • Clean Naming Convention: The name of the layer will be the exact name of the downloaded file. Use lowercase, hyphenated names.
    • ❌ Bad: Rectangle 4323 or Vector
    • ✅ Good: hero-background-image, icon-arrow-right, logo-footer
  • Flatten Icons: Outline strokes (Shift + Cmd + O) and flatten (Cmd + E) complex icons into a single shape layer where possible to ensure clean SVG code.

6. Component Architecture

  • Use Components: Any element that repeats (Buttons, Cards, Navbars, Inputs) must be a Main Component.
  • Component Properties: Use Variants, Boolean, and Text properties cleanly. Our system maps Figma variants directly to CSS modifier classes.

    e.g., A button component with properties Variant: Primary / Secondary and State: Default / Hover.

7. Semantic Design — The Layer Designers Often Skip

This is the chapter most Figma files fail at — and the one with the highest impact on the final product.

Our pipeline reads your Figma file as a source of truth. If a text element is labeled H3 in Figma but visually looks like a page title, our agents will code it as an H3 — not an H1. The visual appearance and the semantic role must match. When they don’t, the resulting code can silently break SEO, accessibility, and screen reader compatibility.

What is Semantic Design?

Semantic Design means every element in Figma is named and structured according to its function and role in the content hierarchy — not just its visual appearance. A “big bold headline” and an “H1” are not the same thing. One describes how it looks. The other describes what it is and where it belongs in the structure of a document.

Search engines, screen readers, and our AI coding agents all rely on this layer of meaning to correctly interpret your design. When it’s missing or inconsistent, the final code produces pages that can rank poorly, fail accessibility audits, and require expensive post-production corrections.

The Heading Chain — The Most Common Failure Point

HTML defines six levels of headings: H1 through H6. Each page should have exactly one H1 (the main topic of the page), followed by a logical hierarchy of H2s (sections), H3s (sub-sections), and so on. Skipping levels or using heading levels purely for size creates an incoherent document structure.

  • One H1 per page: It represents the primary topic. There must be only one. Design accordingly — even if multiple large text elements exist on the page, only one can hold the H1 semantic role.
    • ✗ Bad: Hero headline labeled Heading/Display + section title also labeled Heading/Display
    • ✓ Good: Hero headline labeled Heading/H1 · Section title labeled Heading/H2
  • Don’t confuse visual size with semantic weight: An H5 can be styled at 12px. An H2 can be styled at 80px. The heading level communicates hierarchy to machines — CSS communicates size to humans. Our build agents can apply any font size to any heading level. But they cannot guess the intended hierarchy from a visual design alone.
    • ✗ Bad: A section subtitle styled at 30px labeled H3 — but the only reason it’s H3 is because that size looked right in Figma
    • ✓ Good: That same subtitle correctly labeled H3 because it is a third-level section title, with font-size: 30px applied as a visual override
  • No heading gaps: Do not jump from H2 to H4. Search engines and accessibility tools interpret heading gaps as structural errors.
    • ✗ Bad: H1 → H3 → H5 (gaps in the chain)
    • ✓ Good: H1 → H2 → H3 (continuous, logical hierarchy)

Functional Layer Naming — What We Actually Read

Our extraction agents identify element roles from their Figma layer names and Text Style assignments. A non-semantic layer name like Text or Frame 23 forces the agent to make assumptions — which introduces errors.

  • Text Styles must declare semantic level: Use names like Heading/H1, Heading/H2, Body/Standard, UI/Label, UI/Caption, UI/Tagline. Styles named by size (Text/48px, Bold Large) have zero semantic value.
    • ✗ Bad: Title Bold, Body Copy, Subtitle Medium
    • ✓ Good: Heading/H1, Body/Standard, Heading/H3, UI/Tagline
  • Frame names must describe purpose: Name frames by their semantic role, not their position.
    • ✗ Bad: Frame 84, Group 12, Rectangle
    • ✓ Good: section-hero, card-pricing, nav-primary, footer-links

Beyond Headings — Lists, Tables, Forms & Media

HTML has native elements for virtually every kind of content. When designers use generic text blocks where a specific element is semantically correct, the code degrades in quality.

  • Lists: If content is a collection of items (Features, Steps, Benefits), design them as a list component — not as stacked text blocks. Our agents will output <ul> or <ol> accordingly.
  • Data Tables: Comparative or tabular data (pricing tiers, specification sheets, ROM comparisons) must be structured as tables in Figma — with rows, columns, and clearly named headers. A text-mimicking-a-table is not a table.
  • Forms: Input fields must have associated visible or described label text. If a field has only a placeholder, flag it explicitly. Our build will generate a visually hidden label for accessibility compliance.
  • Blockquotes & Testimonials: Quoted or attributed text must use a dedicated component or Text Style named UI/Quote or UI/Testimonial — not just italic body text.
  • Images & Icons: Every image and meaningful icon must have a layer name that serves as its alt description. Purely decorative elements must be named with a decorative- prefix so agents suppress the alt attribute.
    • ✗ Bad: Image, Photo, SVG 1
    • ✓ Good: anthros-chair-side-view, icon-checkmark-cyan, decorative-background-blob

The Business Case — Why This Matters to Your Results

  • 🔍 SEO: Google’s crawler reads heading hierarchy as a content outline. A broken H1 chain or missing heading structure directly impacts organic ranking potential.
  • ♿ Accessibility: Screen readers navigate by heading levels, list structure, and form labels. Non-semantic design produces pages that are legally and ethically inaccessible to users with disabilities — a growing compliance risk in most markets.
  • ⚡ Build velocity: Every semantic ambiguity in the Figma file generates a clarification round-trip between the agent and the designer. A semantically clean file eliminates this friction entirely and cuts production time significantly.
  • 📐 Maintainability: Code generated from a semantic design is structured and predictable. It’s easier to extend, modify, and debug for years after delivery.

The Rule of Thumb: Before handoff, do a pass through your Figma file asking: “What HTML element would this be?” for every element. If you can’t answer it clearly, the design is not semantically complete.
